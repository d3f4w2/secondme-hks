## 一、官方硬性要求

### 1. 时间线

- `3 月 16 日 19:30`：直播开场宣讲会，正式开赛
- `3 月 20 日 12:00`：滚动报名阶段结束
- `3 月 20 日 24:00`：项目提交截止
- `3 月 21 日 14:00`：公布决赛入围名单
- `3 月 21 日 16:00`：决赛路演与结果公布

### 2. 提交入口

- 官方项目提交页：`https://reconnect-hackathon.com/`

### 3. 项目提交的最低门槛

必须满足：

- 项目通过 `Second Me 开发者平台` 创建应用
- 项目使用官方技术栈开发
- 项目必须接入 `Second Me OAuth` 授权登录
- 项目必须是 `A2A 应用` 或面向 Agent 的应用
- 不能只是普通的人对人 AI 工具
- `Agent 必须是应用的核心参与者`
- 必须提供 `可访问的线上 Demo`
- 仅提交代码仓库 `不参与评审`

### 4. 赛道

提交时必须选择一个赛道：

1. `把互联网重做一遍`
2. `Agent 的第三空间`
3. `无人区`

### 5. 组队规则

- 支持单人或团队参赛
- 每队上限 `4 人`

### 6. 决赛入围规则

- 共 `15 支队伍` 进入决赛路演
- 前 `10 名`：按 `OAuth 授权登录用户数` 排名
- 知乎特别奖前 `5 名`：在申报知乎特别奖的项目中，按 `OAuth 授权登录用户数` 排名

这意味着：

- `用户数` 不是加分项，而是核心晋级条件

### 7. 知乎特别奖条件

满足以下任一条件即可额外申报：

- 用到知乎提供的任意一项能力
- 项目匹配“人类社区重新定义连接”的主题

知乎开放能力：

- `知乎圈子`
- `知乎热榜`
- `全网可信搜`
- `刘看山 IP 授权`

## 二、评审与拿分重点

### 1. 总分结构

- `50%`：评委评分
- `50%`：最终 AI 用户数（OAuth 授权登录数）

### 2. 评委评分维度

- `40%`：A2A 场景价值
- `30%`：创新度
- `30%`：完成度

### 3. 对开发策略的直接含义

这届比赛不是比谁做了一个更精致的 AI 页面，而是比谁同时满足下面四件事：

- `Agent 真的在场`
  - 不能只是“用户点一下，AI 回一句”的普通工具
- `人与人真的被重新连接`
  - 最终要能解释“为什么这两个人值得认识”
- `产品能上线，能跑，能演示`
  - 没有线上 Demo 就没有评审资格
- `要尽早拿用户`
  - 因为 OAuth 用户数同时决定分数和晋级概率

### 4. 现在做项目时要优先保证的事

按优先级看：

1. `Second Me OAuth` 真实可用
2. `线上 Demo` 可访问
3. `A2A 叙事` 足够强
4. `真实用户进入路径` 清楚
5. 如果要冲知乎特别奖，再接入知乎能力

## 三、当前项目对照表：Agent Party

### 1. 当前项目一句话

让多个 Agent 围绕一个话题先开一场小型 party，再把最同频、最值得认识的人从场里捞出来。

### 2. 当前项目更适合哪个赛道

当前最适合：

- `赛道二：Agent 的第三空间`

原因：

- 它本质上是在造一个 Agent 愿意进入的讨论场
- 连接是先在场里自然发生，再被系统提炼出来

### 3. 已满足或已具备的部分

- 已有 `Web GUI` 形态
- 已有 `Second Me OAuth` 项目结构
- 已有 `A2A` 叙事
- 已有“从场里捞出关系”的产品故事
- 方向上符合 `第三空间` 赛道

### 4. 还未真正验证的部分

- `OAuth` 是否真实完整跑通
- 是否已完成 `线上部署`
- 线上回调地址是否已经改成正式可用地址
- 演示时是否能用真实账号顺畅完成授权
- 当前 Agent 交互是否足够让评委认定为“真的 A2A”

### 5. 当前项目的主要风险

- 当前多 Agent 讨论仍偏 `演示式`
  - 更像预设结果，而不是强自主交互
- 还没有真实用户增长路径
  - 但比赛用户数权重很高
- 还没有接知乎能力
  - 不利于冲知乎特别奖
- 还没有最终线上 Demo 版本
  - 这是硬门槛

### 6. 当前项目如果不重做，最少必须补什么

- 跑通真实 `OAuth`
- 完成线上部署
- 明确让评委看得懂的 `A2A 过程`
- 明确让用户看得懂的“为什么是这个人”
- 设计至少一条 `拉 OAuth 用户` 的路径

## 四、候选方向

这部分不是结论，只是直播后用于比较的备选池。

统一评估口径：

- 一句话定义
- 对应赛道
- A2A 强度
- 知乎特别奖适配度
- 3 天 MVP 难度
- 用户增长潜力
- 主要风险

### 方向 A：Agent Party

一句话定义：

- 多个 Agent 围绕同一话题先碰撞，再从场里捞出最值得认识的人

对应赛道：

- `赛道二：Agent 的第三空间`

A2A 强度：

- `高`

知乎特别奖适配度：

- `中`
- 接入热榜后可升到 `高`

3 天 MVP 难度：

- `低到中`

用户增长潜力：

- `中`
- 好处是故事清楚，坏处是用户裂变路径还不够强

主要风险：

- 如果 Party 过程太像脚本，评委会觉得 A2A 不够真
- 如果没有接真实社区语境，知乎特别奖优势不明显

### 方向 B：热点同频局

一句话定义：

- 基于知乎热榜，让多个 Agent 围绕实时热点快速讨论，再从中匹配最同频的人

对应赛道：

- `赛道二`
- 也可兼容 `知乎特别奖`

A2A 强度：

- `高`

知乎特别奖适配度：

- `高`

3 天 MVP 难度：

- `中`

用户增长潜力：

- `高`
- 因为热点天然更容易传播，更适合做分享和拉新

主要风险：

- 需要接知乎热榜接口
- 实时话题会提高产品复杂度
- 如果只做“看热榜 + 配对”，容易浅

### 方向 C：知乎圈子里的 Agent 社交实验

一句话定义：

- 让 Agent 在真实圈子里读帖、发帖、评论和互动，再把真正同频的人连接出来

对应赛道：

- `赛道二`
- 强适配 `知乎特别奖`

A2A 强度：

- `很高`

知乎特别奖适配度：

- `很高`

3 天 MVP 难度：

- `中到高`

用户增长潜力：

- `高`
- 因为真实社区场景更容易讲故事，也更可能被知乎侧看中

主要风险：

- 接口接入和权限申请更重
- 社区动作比单站内 Demo 更复杂
- 容易在时间上失控

### 当前判断

如果只看 `3 天能做完 + 不容易翻车`：

- `方向 A` 最稳

如果看 `知乎特别奖 + 创新感 + 社区语境`：

- `方向 B` 和 `方向 C` 更强

如果直播后官方明确鼓励“真实知乎语境里的 Agent 行为”：

- 优先重新评估 `方向 B / C`

## 五、直播后待确认事项

今晚直播重点关注这些信息：

### 1. 官方对 A2A 的认定边界

重点判断：

- 演示型多 Agent 过程是否足够
- 还是必须更接近真实 Agent 间自主交互

### 2. 官方对 Demo 的期待

重点判断：

- 更看重 GUI 完整度
- 还是更看重 Agent 行为本身

### 3. 是否鼓励接知乎能力

重点判断：

- 官方是否明显偏向使用知乎接口的项目
- 知乎特别奖的实际竞争强度高不高

### 4. OpenClaw / 龙虾接入的时间和重要性

重点判断：

- 对本次评审是否关键
- 还是只是后续生态加分项

### 5. 用户增长怎么理解

重点判断：

- 官方更看重绝对 OAuth 用户数
- 还是也会看用户活跃或传播路径

## 六、当前阶段结论

当前先不锁死方向。

现阶段最合理的工作方式是：

- 先用这份文档守住官方要求
- 今晚看直播，重点听评审口径和技术接入细节
- 直播后再决定：
  - `继续强化 Agent Party`
  - 还是 `切到热点同频局 / 知乎圈子实验`

在直播前，唯一不能搞错的底线是：

- 要有 `线上 Demo`
- 要有 `Second Me OAuth`
- 要让 `Agent` 成为核心参与者
- 产品最后要能解释：`为什么这两个人值得被连接起来`
