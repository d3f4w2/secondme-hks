# 2026-03-16 产品方向文档（草案）

这份文档用于固定当前已经达成一致的产品方向，作为后续 PRD、页面设计和 MVP 开发的上游依据。

当前只记录已经确定的部分，不提前锁死后续功能，不展开实现细节。

## 产品一句话

`找谁` 是一个“有事找谁”的判断助手。

当用户遇到一件现实中的事，不知道该听谁的、该找谁问时，让自己的 Agent 先替他判断。

## 服务对象

第一版服务对象是普通人。

- 他们遇到现实问题、专业争议或热点判断时，不知道谁说得靠谱。
- 他们不缺信息，缺的是更低成本的可信判断和下一步建议。
- 他们平时会在知乎看观点、看回答者、看谁更专业，但这个过程很耗精力，也容易误判。

## 核心价值

- 帮用户从“信息过载”走到“明确下一步”。
- 先回答“听谁的”，再推进到“找谁问”。
- 借助知乎的社区语境，帮助用户判断什么样的人更值得信。
- 让 Agent 真正参与判断过程，而不是只给一段泛泛的 AI 总结。

## 产品定位

`找谁` 不是热点总结工具，不是社交匹配产品，也不是泛 AI 助手。

它是一个现实问题判断助手，核心任务是：

- 判断这类问题该听哪类人。
- 判断哪类人虽然声音大，但不该优先信。
- 给出下一步该找谁问的建议。

## 知乎的角色

第一版里，知乎扮演“可信判断上下文”。

- 系统借助知乎的话题语境、回答者长期表达和社区信任感，判断什么样的人更靠谱。
- 第一版不要求直接在知乎内完成连接动作。
- 第一版也不承诺精确找到某个真实账号，优先做到“找对人群类型”。

## Agent 的角色

第一版里，Agent 是中介机制，不是最终目的。

- 用户进入系统后，先生成一个代表当前问题的“临时办事代理”。
- 这个代理再和其他 Agent 协作，完成判断。
- 当前建议保留 4 个基础角色：`问题识别 Agent`、`可信度判断 Agent`、`风险过滤 Agent`、`行动建议 Agent`。
- `问题识别 Agent`：判断用户遇到的是什么类型的问题。
- `可信度判断 Agent`：判断这类问题应该优先听哪类人。
- `风险过滤 Agent`：指出哪些人看起来很有说服力，但不应优先信。
- `行动建议 Agent`：把判断推进成“接下来该找谁问”。

## 产品名称

当前工作名暂定为 `找谁`。

选择这个名字的原因：

- 足够短，一眼就能懂产品是干什么的。
- 既能覆盖“听谁的”，也能覆盖“找谁问”。
- 比纯观点类名字更有行动感，比纯社交类名字更有真实价值。

当前保留的备选名称：

- `该问谁`
- `听谁`

如果后续产品气质需要更锋利或更品牌化，再单独调整名称。

## 第一版 MVP 场景

第一版只做一个主场景：`听谁的 -> 找谁问`

用户面对一个现实问题、专业争议或热点判断时，不知道该信谁，也不知道接下来该问谁。

系统的输出不是“帮你总结一下”，而是：

- 这类问题应该优先听哪 3 类人。
- 哪 1 到 2 类人不建议优先信。
- 每类判断背后的理由是什么。
- 下一步应该找谁问。

## 第一版演示切口

第一版 demo 推荐聚焦在“热点中的专业判断”。

原因很简单：

- 这个切口最容易和知乎社区语境挂上钩。
- 用户一眼就能理解“听谁的”这个需求。
- 既能体现内容判断，也能自然推进到“找谁问”。
- 比直接做“找谁办”更稳，更适合 3 天内做出清楚的 MVP。

建议 demo 用一句问题驱动，例如：

- “这个热门建议到底该不该信？”
- “这件事我应该先听谁的？”
- “遇到这个问题，我下一步该找谁问？”

## 第一版最小用户链路

1. 用户输入一个真实问题。
2. 系统生成一个代表用户当前任务的临时办事代理。
3. 多个 Agent 对这个问题做分工判断。
4. 结果页输出“该听谁”“不该先信谁”“下一步找谁问”。

## 第一版边界

为了保证 MVP 可落地，第一版先不做这些内容：

- 不做完整的办事中介平台。
- 不做精确到真实知乎账号的全量匹配。
- 不做知乎内直接私信、评论或连接动作。
- 不做多个大场景并行展示。

## 可扩展方向

这版方向文档只固定当前主线，后续可往下扩展，但现在不展开设计：

- 从单一切口扩展到更多现实问题场景。
- 从“找哪类人”扩展到“辅助发起连接”。
- 从知乎的可信判断上下文，扩展到更深的知乎能力接入。

当前默认以这条主线继续推进：

`有事找谁 -> 听谁的 -> 找谁问`
